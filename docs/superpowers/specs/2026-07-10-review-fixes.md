# Spec: hardening-реліз плагіна antigravity (review-fixes, ціль 1.0.2)

Дата: 2026-07-10
Джерело: `docs/superpowers/specs/2026-07-10-review-fixes-design.md` (узгоджений брейншторм)
Скоуп: єдиний пакет `plugins/antigravity/**` + доки в корені. Монорепо нема.

## 1. Проблема

Повне ревью плагіна (рантайм `plugins/antigravity/scripts/**`, хуки, команди, скіли, промпти, схема, доки; 74/74 тести зелені; `claude plugin validate` PASS) виявило 2 критичні, ~8 середніх і ~10 дрібних дефектів. Спільна риса критичних — розрив між тим, що плагін **обіцяє** (валідний JSON зі схеми, read-only ізоляція, дієві прапорці) і тим, що **робить**. Дрібні — накопичений борг (мертвий код, typos, неконсистентна атрибуція, метадані). Ціль фічі — закрити **всі** знахідки одним hardening-релізом 1.0.2; обсяг фіксований, деталі імплементації уточнюються.

Джерело правди про поведінку `agy` 1.0.10 — `docs/agy-cli.md` (live probe): `--sandbox` існує; `-p` бере промпт лише з argv; `--print-timeout` advisory; RESOURCE_EXHAUSTED видно лише в `--log-file`.

## 2. Підхід

Спиратися на наявні механізми, нічого не вигадувати:
- Локи — той самий **O_EXCL**-патерн, що вже в `lib/job-slots.mjs` (retry ~50ms, ліміт ~5s, stale-reclaim по mtime+dead-pid). Переносимо на `state.lock`. PID писаря вписується в lock-файл **атомарно в момент створення** (payload передається у той самий `openSync(..., "wx")`→`writeSync`→`closeSync` до того, як інший процес зможе прочитати файл), а не окремим пізнішим `write()`. Інакше конкурентний reclaim-читач прочитає порожній/недописаний PID і хибно визнає живий лок мертвим. Реюз перевіреного `job-slots.mjs`-хелпера (там ця деталь уже коректна) — не власна копія.
- Каскад таймаутів — через **env-fallback**. `ANTIGRAVITY_COMPANION_TURN_TIMEOUT_MS` у коді **зараз відсутній** — `runOneShot` передає лише `options.timeoutMs`, тож саму читалку env треба **додати** (не «вже вміє»): валідований парсинг env (Number.isFinite, >0, інакше ігнор із logNote) як fallback до `options.timeoutMs`, який виставляє **внутрішній** agy-turn timer на 780s. Без цієї зміни inner timer лишиться 900s і spawnSync уб'є wrapper на 840s раніше за внутрішній BLOCK — початкова гонка повернеться. Обов'язковий інтеграційний тест із fake-agy, що доводить порядок kill-ів (780<840<900).
- Ізоляція — реальний прапорець `agy --sandbox` замість самопроголошеної «read-only».
- Effort — мапінг на **суфікс назви моделі** (`(Low|Medium|High)`), бо в agy effort закодовано саме там.
- Стан — стандартний XDG-ланцюг замість `/tmp`.
- Кожна знахідка супроводжується юніт- та/або argv-тестом (fake-agy fixture розширюється перевірками argv).

Порядок імплементації: спершу критичні (A1, A2), далі середні (B*), тоді дрібні (C*), наприкінці `bump-version 1.0.2` + `check-version`. Гілка A2 (sandbox всюди vs лише inline-diff) обирається ручним пробом з реальним agy на етапі імплементації; дефолт дизайну — sandbox для всіх не-write запусків.

## 3. Зміни

### 3.1 Data model / стан (`lib/state.mjs`)

- **Директорія стану (B4)**: ланцюг `CLAUDE_PLUGIN_DATA` → `XDG_STATE_HOME/antigravity-companion` → `~/.local/state/antigravity-companion`. Прибрати `os.tmpdir()`-фолбек. `mkdirSync` з `mode: 0o700`. Змінюється `resolveStateDir` (`FALLBACK_STATE_ROOT_DIR`).
  - **Одноразова міграція (обов'язкова, не «пропускаємо»)**: тезу «job-и короткоживучі» відкинуто — вона ігнорує `state.config` (зокрема `stopReviewGate`) та живі job handles. Простий перехід на новий шлях **тихо втратив би явно ввімкнений security-gate** (новий XDG-стан стартує з `stopReviewGate=false`) і зробив би активні job-и невидимими для `status`/`cancel`/SessionEnd, лишивши їх працювати як сироти. Тому при першому `loadState` з новим шляхом: якщо новий `state.json` відсутній, а старий /tmp-стан читабельний — **імпортувати щонайменше `config` (весь, з `stopReviewGate`) і незавершені job handles** (running/queued), записати їх у новий стан під локом, і **позначити старий файл як мігрований** (перейменувати в `state.json.migrated` або видалити) щоб міграція була ідемпотентна й не спрацьовувала двічі. Corrupt/нечитабельний старий стан → пропустити міграцію з logNote (не падати). Зафіксувати міграцію в CHANGELOG.
- **Блокування запису (B5)**: новий `state.lock` (O_EXCL, патерн job-slots) береться **лише на write-точках входу і без вкладення**. `updateState` виконує read→mutate→write повністю під одним локом, викликаючи всередині **приватний `saveStateUnlocked`** (без власного лока); публічний `saveState` (для прямих викликів) бере той самий лок один раз навколо свого write. Лок ніколи не вкладається сам у себе (O_EXCL нереентрантний — вкладення = самодедлок/EEXIST). Запис `state.json` — **атомарний write-and-rename**: серіалізувати в тимчасовий `state.json.<pid>.tmp` у тій самій директорії, `fs.renameSync` поверх цільового (rename атомарний на локальній FS). Прямий `fs.writeFileSync` у цільовий файл прибрати. Захищає від конкурентних писарів: background-worker, `/antigravity:cancel`, session-хук.
  - **`loadState` fail-closed (не fail-open)**: default дозволено **лише для ENOENT** (файл ще не існує). Будь-яка інша помилка (`JSON.parse` corrupt, EIO, EACCES, EMFILE) **не** повертає `defaultState()` — вона **кидається/пробрасується**, щоб наступний `updateState`/`saveState` **не перезаписав потенційно валідний файл дефолтом** і не стер історію job-ів. Атомарний rename закриває лише partial-read від конкурентного writer-а — цей окремий fail-open клас має закриватись саме забороною default на parse/I/O errors. Виклики, що читають стан для мутації, при не-ENOENT помилці **abort-ять write** (лишають файл як є) з logNote.
  - **Прямі write-и в `jobs/<id>.json`**: лочити треба **не лише `state.json`**. Per-job файли досі пишуться напряму `writeFileSync` з кількох процесів (progress-апдейт worker-а, completion, `cancel`) → можливий обрізаний JSON / `JSON.parse` failure / `completed` після `cancelled` / розбіжність між job-файлом і `state.json`. Фікс: (а) кожен запис per-job файлу — теж **atomic write-and-rename** під **per-job локом** (`jobs/<id>.lock`, той самий O_EXCL-хелпер), і (б) переходи статусу — **CAS/монотонні** (напр. `cancelled`/`completed`/`failed` — термінальні; термінальний статус не перезаписується не-термінальним; `completed` не затирає `cancelled`). Альтернатива, якщо реалізаційно простіше: зробити `state.json` **єдиним canonical source of truth** для статусів, а `jobs/<id>.json` — похідним/append-only логом без гонок за статус. Вибір гілки — на етапі імплементації; вимога — усунути незалочений read-modify-write per-job файлу.
- **SessionEnd не тримає stale snapshot через kill/rollback (B5/B3)**: teardown робить `loadState`, потім **довгий** kill+rollback, і якби наприкінці викликав `saveState` зі **старим** snapshot — атомарно затер би дані, що worker/інша сесія записали між read і write (lock чесно взявся б, але поверх свіжого стану лягла б stale-копія). Тому фінальний запис teardown — через **`updateState`** (read-modify-write **під локом на свіжому стані**): наново прочитати стан і **відфільтрувати лише ті job id, які teardown реально прибрав**, не переносячи весь застарілий snapshot. Snapshot, знятий до kill, використовується **тільки** щоб знати, кого вбивати, а не як тіло майбутнього запису.
- Формат `state.json` (jobs[]) **не змінюється** — лише механіка доступу.

### 3.2 API / argv- та prompt-контракт

Точка збірки argv — `buildAgyArgs`; виконання — `executeTaskRun`/`runOneShot` (`lib/antigravity.mjs`); CLI-парсинг — `antigravity-companion.mjs`.

- **A1 — схема в промпт + runtime-валідація (fail-closed)**: у `prompts/adversarial-review.md`, блок `<structured_output_contract>`, додати плейсхолдер `{{OUTPUT_SCHEMA}}`; `buildAdversarialReviewPrompt` інтерполює `JSON.stringify(schema, null, 2)` з `schemas/review-output.schema.json` (включно з top-level `next_steps`). Прибрати мертвий `outputSchema: readOutputSchema(REVIEW_SCHEMA)` з виклику `runTurn` (runOneShot його ігнорує).
  - **Схема в промпті НЕ гарантує schema-valid output** — поточний `parseStructuredOutput` (`lib/antigravity.mjs:829`) робить лише `JSON.parse`, а renderer перевіряє кілька top-level полів, тож невалідні `verdict`/`severity`/`confidence`, зайві поля чи неповні `findings` пройдуть як «успіх». Тому додати **runtime JSON-Schema валідацію** розпарсеного output проти `schemas/review-output.schema.json` (легкий валідатор без мережевих залежностей; той самий файл-схема, що йде в промпт). Результат **fail-closed**: невалідний output — це помилка ревью, а не успіх.
  - **Bounded repair/retry**: при провалі валідації — щонайбільше **один** repair-turn (передати agy назад помилки валідатора з вимогою повернути валідний JSON); якщо і він невалідний — повернути явну помилку (не мовчазний «успіх» з поламаними даними). Ліміт retry жорсткий, щоб не зациклити turn-бюджет.
- **A2 — sandbox**: `buildAgyArgs` додає `--sandbox` для не-write запусків (review, adversarial-review, task без `--write`). Не додає для `task --write`. Escape hatch: env `ANTIGRAVITY_COMPANION_NO_SANDBOX=1` вимикає. `--dangerously-skip-permissions` лишається (без нього headless-agy висне) — це чесно документується.
  - **`--sandbox` НЕ приймати як read-only boundary без доказу**: локальний probe підтверджує лише terminal-обмеження, а не заборону workspace-write, symlink-write, non-terminal tools чи доступ поза repo; `--dangerously-skip-permissions` при цьому автоапрувить усі tools. Тому потрібні **canary-тести реального agy** (ручний пост-мердж проб + зафіксований чек-лист у README «Security posture»): під `--sandbox` спробувати (1) write у **tracked** файл, (2) write у **untracked** файл, (3) write **поза workspace**, (4) **symlink-escape** (запис через symlink, що вказує назовні). Кожен має бути **заблокований**. Якщо хоч один проходить — `--sandbox` не є ізоляційним boundary, і README/доки мають це чесно відобразити (не обіцяти read-only), а рішення по A2-гілці (sandbox всюди vs лише inline-diff) переглянути. Запланований тест «self-collect працює під sandbox» перевіряє доступність, а **не** блокування модифікацій — обидва аспекти обов'язкові.
- **B1 — effort → модель**: значення звузити до `low|medium|high` (інше → помилка з підказкою). Базова модель = задана (аліас/label) або дефолт `Gemini 3.5 Flash`; суфікс `(Low|Medium|High)` підставляється/замінюється за effort. Якщо label не має суфікс-патерну (напр. `Claude Sonnet 4.6 (Thinking)`) — чітка помилка «effort не застосовний до цієї моделі». Прапорець доходить до agy через `--model "<label> (High)"`.
- **B2 — таймаути**: (а) хук читає `input.stop_hook_active === true` → logNote + allow (жодного повторного ревью в тому ж stop-циклі); (б) каскад: agy-turn 780s < spawnSync 840s < hook 900s. Хук передає ліміт у task через env `ANTIGRAVITY_COMPANION_TURN_TIMEOUT_MS`. **Ця env-читалка в коді зараз відсутня** — `runOneShot` передає лише `options.timeoutMs`. Тому реалізувати треба **обидві** сторони: (1) хук виставляє env, (2) `runOneShot` **додає** валідований парсинг цього env (Number.isFinite, >0; інакше ігнор із logNote) як fallback до `options.timeoutMs`, і цей ліміт застосовується до **внутрішнього** agy-turn timer (780s). Реалізувати лише env у хуку **недостатньо**: inner timer лишиться 900s, spawnSync уб'є wrapper на 840s раніше за внутрішній BLOCK — гонка не зникне. Обов'язковий інтеграційний тест із fake-agy, що доводить фактичний порядок kill-ів (внутрішній 780 спрацьовує перший). `hooks.json` Stop timeout лишається 900.
- **B3 — SessionEnd teardown**: `hooks.json` SessionEnd timeout 5 → 60; у teardown `terminateProcessTree(pid, { graceMs: 2000 })`.
  - **Прибрати руйнівний auto-rollback**: поточний teardown після kill викликає `restoreWorkspaceSnapshot` (`lib/git.mjs:350`), яка при незмінному HEAD робить `git reset --hard` — це **безповоротно стирає всі tracked-edits**, що користувач/інший агент зробив після зняття snapshot, разом зі змінами agy. Збільшення timeout 5→60s лише **підвищує шанс**, що деструктивний rollback встигне добігти. Тому: (1) **не робити auto `git reset --hard`** при невідомих post-snapshot змінах у tracked-файлах — якщо working tree розійшовся зі snapshot непередбачувано, teardown лишає його як є і логує; (2) поточний tree перед будь-яким відкатом зберігати у **recoverable** формі — `git stash create`/`stash store` або окремий unreachable-commit — щоб зміни завжди можна було відновити (`git stash list`/reflog), а не втратити; (3) стратегічно — write-задачі виконувати в **окремому worktree**, щоб teardown не чіпав основний workspace користувача. Мінімум для цього релізу — прибрати беззастережний `reset --hard` і зробити відкат recoverable; worktree-ізоляцію зафіксувати як напрям.
- **B6 — self-collect untracked**: у гілці `includeDiff === false` (`git.mjs`) інлайнити лише список імен+розмірів (перші ~200, далі «…and N more»), без тіл файлів.
- **B7 — Windows shell (command injection)**: `shell:true` **не можна лишати** для Git/native-бінарників. `runCommand` отримує user-controlled `--base <ref>` і передає його Git-командам; з `shell:true` cmd.exe інтерпретує `&`, `|`, `>`, `^` тощо, argv **перестає бути security boundary**, і crafted ref виконає другу команду (Windows command injection). Тому: **`shell:false` для всіх Git/native-спавнів** (argv лишається межею на всіх ОС). Проблему `.cmd`/`.bat` (git/npm/обгортки agy у PATH як `.cmd`), через яку `spawnSync` з `shell:false` дасть ENOENT, розв'язувати **явним резолвом розширення**, а **не** shell-ом: на win32 резолвити повний шлях бінарника через `PATHEXT` (перебір `.exe`/`.cmd`/`.bat`/`.com`) або `where.exe`, і спавнити **резолвлений шлях** з `shell:false`. Аргументи (включно з `--base <ref>`) при цьому ніколи не проходять через інтерпретатор команд. Прибрати і поточний небезпечний `process.env.SHELL || true` (POSIX-`SHELL` з git-bash перехоплював би win32-спавни), і `shell:true`. Для `binaryAvailable` на win32 — резолв через той самий PATHEXT/`where.exe`-механізм (напр. знайти `npm.cmd`), теж без shell.
- **C1 — auth-probe cwd**: `getAntigravityAuthStatus` виконувати з одноразового `mkdtemp`-каталогу (не з workspace), щоб не перетирати `last_conversations.json[cwd]` і не ламати `--resume-last`.
- **C8 — splitRawArgumentString бекслеші** (`lib/args.mjs`): бекслеш екранує лише наступні пробіл/лапку/бекслеш; перед іншими символами — literal (`C:\Users\x` виживає, `\"` досі екранує).
- **C9 — `--timeout-ms 0`** (`antigravity-companion.mjs`): замінити `|| DEFAULT` на `Number.isFinite`-перевірку; 0 = миттєвий снапшот без очікування.

## 4. Edge-cases

- **A2**: `--sandbox` може заблокувати self-collect git-інспекцію в якійсь версії agy → escape env + fallback-гілка (sandbox лише для inline-diff). Рішення — ручний проб з реальним agy.
- **B1**: модель без суфікс-патерну (`(Thinking)`, кастомний label) — не мовчати, а помилка. Аліас має резолвитись у label ДО підстановки суфікса.
- **B2**: `stop_hook_active` відсутній/не-true → нормальний шлях; таймаути мають лишатись строго впорядкованими (780<840<900), інакше зовнішній kill знову випередить внутрішній BLOCK.
- **B4**: ні `XDG_STATE_HOME`, ні `HOME` → визначити поведінку (напр. помилка чи локальний фолбек); тест через тимчасові HOME/XDG.
- **B5 lock**: stale-lock (мертвий писар лишив `state.lock`) — reclaim по mtime>10s + мертвий pid усередині; PID у lock-файл пишеться **атомарно при O_EXCL-створенні**, інакше reclaim-читач прочитає порожній PID і хибно вб'є живий лок.
- **B5 fail-closed**: `state.json` corrupt/EACCES/EIO → `loadState` **не** повертає default (лише ENOENT); write abort-иться, файл лишається.
- **B5 per-job**: одночасні progress/completion/cancel на `jobs/<id>.json` — atomic rename під per-job локом; термінальний статус (`cancelled`/`completed`/`failed`) не перезаписується не-термінальним.
- **B3**: post-snapshot tracked-edit користувача при незмінному HEAD — teardown **не** робить `reset --hard`, відкат recoverable.
- **B4 міграція**: старий стан читабельний → імпорт config+живих job; corrupt старий стан → skip із logNote; повторний старт → без дублювання (ідемпотентність).
- **B7**: `--base` з shell-метасимволами (`& | > ^`) — не виконує другу команду (`shell:false`); на win32 бінарник у PATH як `.cmd` — резолвиться через PATHEXT/`where.exe`, не ENOENT.
- **B6**: >200 untracked — обрізання «…and N more»; 0 untracked — гілка не ламається.
- **C8**: trailing-бекслеш у кінці аргументу; подвійний бекслеш; `\"` всередині лапок.
- **C9**: `--timeout-ms 0` = миттєвий снапшот; від'ємні/NaN — існуюча валідація.
- **C1**: `mkdtemp` cleanup після проби; проба не має лишати сміття.

## 5. Ризики

- **A2 (найбільший)**: два підризики. (1) `--sandbox` реально ламає self-collect → escape env + fallback-гілка (sandbox лише для inline-diff). (2) `--sandbox` **не є** read-only boundary (пропускає workspace/symlink/outside-repo write) → доки хибно обіцяли б ізоляцію. Мітигація обох: **canary-тести реального agy** (tracked/untracked/outside-workspace write + symlink-escape мають бути заблоковані) ПЕРЕД релізом; за провалу — не обіцяти read-only в README і переглянути A2-гілку.
- **B5 локи**: неправильний stale-reclaim → або deadlock (лок ніколи не звільняється), або втрата захисту. Ще один клас — самодедлок від вкладеного лока (`updateState`→`saveState`): усунуто розділенням на залоканий вхід + приватний `saveStateUnlocked`. Мітигація: точна копія перевіреного job-slots патерну + атомарний write-and-rename + стрес-тест 15-20 паралельних писарів.
- **B7 shell на win32 (безпека > зручність)**: `shell:true` на win32 = command injection через user-controlled `--base <ref>` (cmd.exe інтерпретує `&|>^`). Мітигація: `shell:false` скрізь для Git/native-спавнів; `.cmd`/`.bat`-резолв робимо **явним PATHEXT/`where.exe`-lookup повного шляху**, а не shell-ом; наявні тести не деградують (argv лишається межею на всіх ОС). Ризик протилежного напряму (ENOENT на `.cmd` без резолву) закриває саме явний lookup.
- **B3 rollback**: беззастережний `git reset --hard` у teardown → втрата паралельних змін користувача. Мітигація: прибрати auto-hard-reset при невідомих post-snapshot змінах, робити відкат recoverable (stash/unreachable-commit), напрям — write у окремому worktree.
- **B5 fail-open loadState**: default на будь-яку помилку читання → наступний write стирає валідний файл дефолтом. Мітигація: default лише для ENOENT, решта помилок abort-ять write.
- **B1 effort**: звуження допустимих значень — breaking для тих, хто передавав інші рядки. Прийнятно (був no-op), але відзначити в CHANGELOG.
- **bump-version**: розсинхрон 4 файлів версії → `check-version` як гейт наприкінці.

## 6. Верифікація

- `cd <root> && npm test` — 74 наявні + новий юніт/argv-тест на КОЖЕН пункт (fake-agy fixture: `--sandbox` присутній/відсутній, effort→model label, схема в промпті `agy -p`, cwd auth-проби ≠ workspace, порожній spawn при `stop_hook_active`).

- **B2 kill-order інтеграційний тест** (fake-agy, що спить довше за всі ліміти): доводить, що внутрішній 780s спрацьовує раніше за spawnSync 840s і hook 900s; окремо — що `runOneShot` реально читає `ANTIGRAVITY_COMPANION_TURN_TIMEOUT_MS` (без env — дефолт; з валідним env — перекриває inner timer; NaN/≤0 — ігнор).
- **B5 stress + fail-closed тести**: 15-20 паралельних писарів `state.json` та per-job `jobs/<id>.json` без обрізаних/втрачених записів і без термінально→не-термінально переходів; `loadState` при corrupt/EACCES **не** повертає default і **не** дає наступному write стерти файл; SessionEnd-teardown після конкурентного write не затирає свіжий стан (updateState на свіжому read).

- **B3 rollback тест**: post-snapshot tracked-edit користувача не зникає після teardown; відкат recoverable (знаходиться в stash/reflog).
- **B7 injection тест**: `--base` з `& | > ^` не виконує другу команду (argv-межа збережена, `shell:false`); win32-резолв `.cmd` через PATHEXT/`where.exe` знаходить бінарник без shell.

## 7. Поза скоупом (non-goals)

- Нові можливості agy (`--add-dir`, мультируту).
- **Повна** міграція історії старого /tmp-стану. **У скоупі** (B4) — одноразовий імпорт `config` (з `stopReviewGate`) і живих job handles, щоб не втратити security-gate й не осиротити активні job-и; поза скоупом лише перенесення завершеної історії job-ів.
- Повна Windows CI-матриця (лише перехід на `shell:false` для Git/native-спавнів + явний PATHEXT/`where.exe`-резолв `.cmd`/`.bat` без shell).
- Зміна session-scoped моделі job-ів (лише чесні доки — C2).
- Будь-які нові фічі поза списком знахідок.
